# 教材配套资源与勘误发布仓库方案

## 一、背景

教材需要提供配套资源和勘误反馈入口。相关资源可能包括演示幻灯片、作业、参考答案、示例代码、实验指导材料、项目完整工程文件等。

可以先准备好访问地址，内容后续再替换和完善。因此，当前阶段的重点不是一次性整理完所有资料，而是先建立一个稳定、可公开访问、后续可持续维护的资源发布入口。

## 二、总体方案

建议建立一个 GitHub 仓库，定位为：

> 教材配套资源与勘误发布仓库

该仓库承担三项核心功能：

1. 发布教材配套资源；

2. 提供可写入书中和二维码中的稳定访问地址；

3. 承接读者勘误、问题反馈和资源更新说明。

推荐采用：

```Plaintext
GitHub 仓库 + GitHub Pages + GitHub Issues + 反馈邮箱
```

其中：

- GitHub 仓库用于存放和管理资源；

- GitHub Pages 用于生成公开访问页面；

- GitHub Issues 用于收集勘误和读者反馈；

- 反馈邮箱作为不熟悉 GitHub 的读者的备用反馈渠道。

## 三、优势

该方案适合当前阶段，原因如下：

1. 地址稳定
书中可以放 GitHub Pages 地址或仓库地址，后续资源内容更新时不需要修改书中的地址。

2. 便于维护
资源可以按章节、类型逐步补充，不要求一开始全部完成。

3. 支持勘误
GitHub Issues 可以记录、分类和追踪读者反馈，也便于后续整理公开勘误表。

4. 成本较低
不需要单独开发网站，GitHub 仓库和 Pages 就能满足第一阶段需求。

5. 便于协作
两位助教可以分工整理资料、提交更新、维护文档。

## 四、仓库命名建议

如果教材名称已经确定，建议使用具体名称，例如：


```Plaintext
教材配套资源与勘误
```

## 六、各目录说明

### README\.md

仓库首页，用于说明：

- 本仓库用途；

- 配套资源类型；

- 下载方式；

- 勘误反馈方式；

- 当前维护人员或维护说明。

### docs/errata\.md

公开勘误表，用于记录已经确认的问题，例如：

- 章节位置；

- 页码；

- 原文内容；

- 修改建议；

- 确认状态；

- 更新时间。

### docs/update\-log\.md

资源更新日志，用于记录每次资源变更，例如：

- 新增某章 PPT；

- 更新某个实验指导；

- 修复某段示例代码；

- 补充某章参考答案。

### docs/guide\.md

读者使用说明，用于解释如何下载资源、如何运行代码、如何提交反馈。

### slides/

存放各章节演示幻灯片。

### code/

存放书中示例代码。

### labs/

存放实验指导材料。

### projects/

存放项目完整工程文件。

### assignments/

存放作业材料。

### answers/

存放参考答案。该目录是否公开需要单独确认，如果不适合完全公开，可以先保留目录，但暂不上传内容。

### \.github/ISSUE\_TEMPLATE/

存放 GitHub Issues 模板，方便读者按统一格式提交勘误或资源问题。

## 七、资源发布原则

1. 先建地址，后补内容
先完成仓库框架、公开页面和二维码地址，后续逐步替换和完善资源。

2. 先放确定可公开的材料
对于版权、答案公开范围、内部教学资料等不确定内容，先不要直接公开。

3. 按章节组织
如果教材章节结构已经确定，建议所有资源尽量对应到具体章节，便于读者查找。

4. 保持文件命名规范
建议使用英文目录和文件名，避免跨平台兼容问题。

5. 大文件谨慎上传
如果 PPT、视频或项目文件过大，可以使用压缩包、Release、网盘或学校课程网站承载，GitHub 仓库中只放说明和链接。

## 八、勘误与反馈机制

建议同时提供两种反馈方式：

1. 邮箱反馈
适合普通读者，门槛低。

2. GitHub Issues
适合熟悉 GitHub 的读者，方便追踪、分类和关闭问题。

GitHub Issues 可以设置两类模板：

- 勘误反馈；

- 资源问题或资源请求。

勘误 Issue 建议包含：

```Plaintext
问题类型：
所在章节：
页码：
原文内容：
建议修改：
问题说明：
```

资源问题 Issue 建议包含：

```Plaintext
资源类型：
所在章节：
问题描述：
期望补充或修改内容：
```

- 处理读者反馈；

- 更新勘误表；

- 定期整理 Issues；

- 发布资源更新日志。
